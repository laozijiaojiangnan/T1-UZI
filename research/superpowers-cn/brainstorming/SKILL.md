# 将想法头脑风暴成设计

## 概述

通过自然的协作对话，帮助将想法转化为完整的设计和规格。

首先了解当前项目上下文，然后一次问一个问题来完善想法。一旦你理解了要构建什么，就以小段（200-300 字）展示设计，每段之后询问到目前为止看起来是否正确。

## 流程

**理解想法：**
- 首先检查当前项目状态（文件、文档、最近的提交）
- 一次问一个问题来完善想法
- 尽可能使用选择题，但开放式问题也可以
- 每条消息只问一个问题 - 如果一个话题需要更多探索，拆分成多个问题
- 专注于理解：目的、约束、成功标准

**探索方案：**
- 提出 2-3 种不同的方案及其权衡
- 以对话方式呈现选项，附上你的建议和理由
- 首先提出你推荐的选项并解释原因

**展示设计：**
- 一旦你认为理解了要构建什么，就展示设计
- 将其分成 200-300 字的小段
- 每段之后询问到目前为止看起来是否正确
- 涵盖：架构、组件、数据流、错误处理、测试
- 如果有些东西不合理，准备好返回并澄清

## 设计之后

**文档：**
- 将验证过的设计写入 `docs/plans/YYYY-MM-DD-<topic>-design.md`
- 如果可用，使用 elements-of-style:writing-clearly-and-concisely skill
- 将设计文档提交到 git

**实现（如果继续）：**
- 询问："准备好开始实现了吗？"
- 使用 superpowers:using-git-worktrees 创建隔离的工作空间
- 使用 superpowers:writing-plans 创建详细的实现计划

## 核心原则

- **一次一个问题** - 不要用多个问题压倒对方
- **优先选择题** - 可能的情况下比开放式问题更容易回答
- ** ruthlessly YAGNI** - 从所有设计中移除不必要的功能
- **探索替代方案** - 在确定之前总是提出 2-3 种方案
- **增量验证** - 分段展示设计，逐段验证
- **保持灵活** - 当有些东西不合理时返回并澄清
