# 将想法转化为设计

## 概览

通过自然协作对话，帮助将想法转化为完整的设计和规格说明。

首先了解当前项目状态，然后一次提一个问题来细化想法。一旦你理解了要构建的内容，将设计分成小块展示（200-300 字），每部分展示后询问是否正确。

## 执行流程

**理解想法：**
- 先了解项目状态（文件、文档、最近的提交）
- 一次提一个问题来细化想法
- 尽量用选择题，开放式问题也可以
- 一次只问一个问题——如果某个主题需要更多探索，拆成多个问题
- 聚焦于：目的、约束、成功标准

**探索方案：**
- 提出 2-3 种不同方案，并说明取舍
- 用对话方式呈现选项，推荐方案并解释原因
- 直接说明你推荐哪个方案以及为什么

**展示设计：**
- 一旦你认为自己理解了要构建的内容，就可以展示设计
- 把它分成 200-300 字的小节
- 每节之后询问「到目前为止看起来对吗」
- 覆盖内容：架构、组件、数据流、错误处理、测试
- 准备好在某些内容不合理时回去澄清

## 设计之后

**文档：**
- 将经过验证的设计写入 `docs/plans/YYYY-MM-DD-<主题>-design.md`
- 如果有 `elements-of-style:writing-clearly-and-concisely` 技能，可以使用它
- 将设计文档提交到 git

**实现（如果继续）：**
- 询问：「准备好开始实现了吗？」
- 使用 `superpowers:using-git-worktrees` 创建隔离工作区
- 使用 `superpowers:writing-plans` 创建详细实施计划

## 核心原则

- **一次一个问题** - 不要一次性问太多问题
- **优先使用选择题** - 比开放式问题更容易回答
- **YAGNI 冷酷无情** - 从所有设计中删除不必要的功能
- **探索替代方案** - 在确定之前总是提出 2-3 种方案
- **增量验证** - 分块展示设计，验证每一块
- **保持灵活** - 当某些内容不合理时回去澄清
