# Plan 命令

此命令调用 **planner** 代理在编写任何代码之前创建全面的实施计划。

## 此命令的作用

1. **重述需求** - 明确需要构建什么
2. **识别风险** - 暴露潜在问题和阻塞点
3. **创建步骤计划** - 将实施分解为阶段
4. **等待确认** - 必须收到用户明确批准后才能继续

## 何时使用

在以下情况使用 `/plan`：
- 开始新功能
- 进行重大架构变更
- 处理复杂重构
- 将影响多个文件/组件
- 需求不明确或模糊

## 工作原理

planner 代理将：

1. **分析请求** 并以清晰的术语重述需求
2. **分解为阶段** 并提供具体、可执行的步骤
3. **识别依赖关系** 在组件之间
4. **评估风险** 和潜在阻塞点
5. **预估复杂度**（高/中/低）
6. **呈现计划** 并等待你的明确确认

## 使用示例

```
用户：/plan 我需要在市场结算时添加实时通知

代理 (planner)：
# 实施计划：实时市场结算通知

## 需求重述
- 当用户关注的市场结算时发送通知
- 支持多种通知渠道（应用内、邮件、Webhook）
- 确保通知可靠送达
- 包含市场结果和用户持仓结果

## 实施阶段

### 阶段 1：数据库架构
- 添加通知表，包含列：id、user_id、market_id、type、status、created_at
- 添加 user_notification_preferences 表用于渠道偏好
- 在 user_id 和 market_id 上创建索引以提高性能

### 阶段 2：通知服务
- 在 lib/notifications.ts 中创建通知服务
- 使用 BullMQ/Redis 实现通知队列
- 为失败投递添加重试逻辑
- 创建通知模板

### 阶段 3：集成点
- 接入市场结算逻辑（当状态变为"已结算"时）
- 查询市场中所有有持仓的用户
- 为每个用户入队通知

### 阶段 4：前端组件
- 在头部创建 NotificationBell 组件
- 添加 NotificationList 弹窗
- 通过 Supabase 订阅实现实时更新
- 添加通知偏好设置页面

## 依赖项
- Redis（用于队列）
- 邮件服务（SendGrid/Resend）
- Supabase 实时订阅

## 风险
- 高：邮件送达率（需要 SPF/DKIM）
- 中：1000+ 用户每市场的性能
- 中：如果市场频繁结算可能导致通知垃圾
- 低：实时订阅开销

## 预估复杂度：中
- 后端：4-6 小时
- 前端：3-4 小时
- 测试：2-3 小时
- 总计：9-13 小时

**等待确认**：继续执行此计划？（是/否/修改）
```

## 重要说明

**关键**：在你明确用"是"或"继续"或类似的肯定回复确认计划之前，planner 代理**不会**编写任何代码。

如果你想要更改，请回复：
- "修改：[你的更改]"
- "不同方法：[替代方案]"
- "跳过阶段 2，先做阶段 3"

## 与其他命令的集成

规划后：
- 使用 `/tdd` 以测试驱动开发方式实施
- 如果出现构建错误，使用 `/build-and-fix`
- 使用 `/code-review` 审查已完成的实现
